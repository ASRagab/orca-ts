## Why

Repo-mutating Orca workflows should not require a perfectly green repository before
they can help. A red baseline test, lint, or verification gate is already a
repairable defect, so the default workflow behavior should repair it before
continuing to later feature or cleanup work.

Dirty worktrees are different: they may contain user-owned work. Those should be
accepted only through an explicit opt-in that snapshots the baseline before any
agent edit.

## What Changes

- Make baseline gate repair the default for authored mutating workflows.
- Keep a clean worktree as the default safety boundary.
- Add an explicit dirty-baseline opt-in for workflows that may repair or build on
  pre-existing local changes.
- Require baseline repair to be bounded, monitored, and validation-preserving.
- Require non-converging baseline repair to fail before downstream workflow stages
  begin.

## Capabilities

### New Capabilities

None.

### Modified Capabilities

- `workflow-skill-authoring`: generated mutating artifacts default to repairing red
  baseline gates before their main work begins.
- `workflow-skill-execution`: flow execution/healing treats red baseline gates as
  an in-flow repair stage by default, while dirty baseline acceptance remains
  explicit and snapshot-backed.

## Impact

- `.orca/workflows/*` and `.orca/loops/*` templates generated by `orca-ts-author`
- `orca-ts-flow` run/diagnosis guidance
- Workflow runbooks and template typecheck gates
- Potential public helper/configuration surface for baseline policy parsing and
  baseline snapshot capture
