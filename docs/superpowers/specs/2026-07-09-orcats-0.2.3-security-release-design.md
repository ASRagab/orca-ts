# Orcats 0.2.3 Security Release Design

## Goal

Publish `@twelvehart/orcats@0.2.3` with a clean development dependency
audit, a working npm CLI entrypoint, and unchanged runtime behavior.

## Evidence

`npm audit` reports three vulnerable development nodes:

`eslint-plugin-boundaries@5.4.0` → `@boundaries/elements@1.2.0` →
`handlebars@4.7.8`.

The critical advisory is `GHSA-2w6w-674q-4c4q`; the same Handlebars version
also carries high, moderate, and low advisories. `npm audit --omit=dev` and a
fresh consumer installation of `@twelvehart/orcats@0.2.2` report zero
vulnerabilities.

Current `main` also deleted the tracked `bin/orcats` shim and ignored `bin/`
without changing the package contract. Clean CI consequently has 15 failing
CLI tests, while a stale ignored local copy can hide the regression.

## Considered Approaches

1. Upgrade `eslint-plugin-boundaries` from v5 to v6.0.2. This is the first
   upstream release that intentionally moves Handlebars to 4.7.9. Preserve
   v5's import-only analysis and migrate the existing rule to supported v6
   syntax. This is the selected approach.
2. Override Handlebars to 4.7.9. This minimizes the direct version change but
   leaves transitive override debt and does not update the owning plugin.
3. Upgrade directly to v7.0.2. This is current, but adds a second major-version
   migration and an entity model that the patch release does not need.

## Changes

- Set `eslint-plugin-boundaries` to `^6.0.2` and regenerate `bun.lock`.
- Set `boundaries/dependency-nodes` to `["import"]` so the lint policy checks
  the same dependency forms as v5.
- Rename `boundaries/element-types` to `boundaries/dependencies` and convert
  its selectors to v6 object syntax without changing the allow/disallow
  matrix.
- Remove `bin/` from `.gitignore` and restore the four-line tracked
  `bin/orcats` Bun shim required by package metadata and CLI tests.
- Bump package, CLI, README, agent notes, and published documentation version
  literals from `0.2.2` to `0.2.3`.
- Exclude the main checkout's self-dependency and npm lockfile experiment from
  the branch; this repository continues to use `bun.lock`.

## Verification

- Confirm the 15 clean-checkout CLI failures become passing tests.
- Run `bun audit --json` and require no advisories.
- Run `bun run lint`, `bun run verify`, and `bun run smoke:package`.
- Inspect the packed manifest and confirm the runtime dependency graph remains
  `effect`, `neverthrow`, and `zod`.
- Obtain an adversarial code review before final validation.
- Merge only after GitHub CI passes.
- Tag the merge commit `v0.2.3`, watch all Release jobs, and verify GitHub
  assets, npm metadata/provenance, CLI version output, and a fresh consumer
  audit.

## Failure Handling

Do not reuse `0.2.3` if npm accepts it and a later release step fails. Fix
forward with the next patch. If npm publication has not occurred, follow the
existing 0.x release runbook for tag and GitHub Release recovery.
